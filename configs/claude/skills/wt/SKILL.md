# Worktree + Draft PR 부트스트랩

새 작업을 시작할 때 워크트리·브랜치·draft PR 을 한 번에 깔아두는 skill 이에요. 코드는 손대지 않아요.

다만 입력의 성격에 따라 PR body 가 달라져요:

- **부트스트랩 모드** — 입력이 "앞으로 뭐뭐 할 거야" 같은 단순 의도 표명이면, 이후 작업이 들어갈 빈 공간만 마련해요. PR body 는 짧은 골격만.
- **스펙/조사 모드** — 입력이 구체적인 계획·명령이거나 "조사해줘 / 설계해줘 / 계획 짜줘" 같은 요청이면, **실제로 그 계획을 세우거나 조사를 수행하고** 결과를 PR body 에 채워요. PR body 가 구현 spec 또는 조사 결과 그 자체가 돼요.

이 두 모드를 [0.5 작업 의도 판별](#05-작업-의도-판별) 에서 먼저 가른 뒤 진행해요.

## 언제 쓰나

사용자가 다음과 같이 호출:

- `/wt {이유}` — 슬래시 커맨드, `{이유}` 는 한국어 또는 영어 자유 문장
- "워크트리 만들고 빈 PR 올려줘", "{이유} 작업 시작할 거야" 같은 자연어 요청

## 입력

`{이유}` 한 줄. 예시:

- `/wt 포인트 결제 환불 기능 추가` — 자유 문장
- `/wt fcm ios 알림 사운드 버그 수정` — 자유 문장
- `/wt #344 키오스크 카페 주문 API` — 자유 문장 + 이슈 번호 힌트
- `/wt refactor supplier query handler` — 영어 자유 문장
- `/wt #123` — **이슈 번호만**. 이 경우 `gh issue view 123` 으로 이슈를 조회해서 제목·본문을 `{이유}` 로 삼아요. 자세한 처리는 [0. 이슈 단독 입력 해석](#0-이슈-단독-입력-해석) 참고.

## 진행 절차

### 0. 이슈 단독 입력 해석

입력이 `#123` / `이슈 123` / 숫자만(`123`) 처럼 **이슈 번호 외의 설명이 거의 없는** 형태이면, 먼저 GitHub 이슈를 조회해서 본문을 `{이유}` 로 만들어요:

```bash
gh issue view {번호} --json number,title,body,labels,state
```

- `state` 가 `CLOSED` 면 사용자에게 "이미 닫힌 이슈예요. 계속할까요?" 짧게 확인한 뒤 진행 (background 세션이라 확인이 불가능하면 그대로 진행하되 PR body 의 참고 사항에 명시).
- `labels` 에서 type 힌트를 우선 활용: `bug` → `fix`, `enhancement`/`feature` → `feat`, `documentation` → `docs`, `refactor` → `refactor`, `chore` → `chore`. 라벨이 없으면 1단계 type 표로 fallback.
- description 은 이슈 `title` 에서 한국어/영어 핵심 명사구를 뽑아 kebab-case 변환. 이슈 본문은 너무 장황하므로 title 우선, 필요 시 본문 첫 문단만 참고.
- 브랜치명에는 **이슈 번호를 반드시 prefix 로** 넣어요: `feat/123-cafe-order-pagination`.
- PR body 의 `## 관련 이슈` 에는 `- 해결할 이슈: #{번호}` 를 자동으로 채워요.
- PR body `## 개요` 첫 문단은 이슈 `title` 을 그대로 한국어로 가져와 한 문장으로 정리해요.

`gh issue view` 가 실패(이슈 미존재 / 권한 없음 / `gh` 미인증)하면 멈추고 사용자에게 보고해요 — 추측으로 진행하지 않아요.

### 0.5 작업 의도 판별

브랜치를 따기 전에, 입력이 어느 모드인지 먼저 판단해요. 판단 결과는 4단계(커밋)와 6단계(PR body)의 분기에 쓰여요.

**부트스트랩 모드** — 아직 무엇을 어떻게 할지 정하지 않은, 의도/주제만 던지는 입력:

- "포인트 결제 환불 기능 추가할 거야", "fcm 알림 버그 고칠 거야 작업 시작"
- 미래형·의지 표현("~할 거야", "~하려고", "시작할게"), 명사구 주제만 있고 구체 지시가 없는 경우
- `#123` 처럼 이슈 번호만 주어졌고 이슈 본문도 한두 줄짜리 주제 수준인 경우
- → 빈 커밋 + 골격 PR body. 실제 조사·설계는 하지 않아요.

**스펙/조사 모드** — 무엇을 어떻게 할지에 대한 구체적 지시이거나, 산출물을 요구하는 명령:

- "결제 환불은 PG 취소 API 호출 → 포인트 원복 → 알림 순서로 처리되게 설계해줘"
- "이 버그 원인 조사해서 정리해줘", "리팩토링 계획 구체적으로 짜줘", "어떤 방식이 나은지 비교 조사해줘"
- 명령형 동사("조사해줘 / 설계해줘 / 계획 짜줘 / 분석해줘 / 비교해줘 / 정리해줘"), 구체적 요구사항·제약·단계가 포함된 입력
- 이슈 본문이 이미 상세한 요구사항/재현 절차/스펙을 담고 있는 경우
- → **실제로 조사·설계를 수행한 뒤** 그 결과를 PR body 에 채워요. (자세한 절차는 6단계 참고)

판별이 애매하면, 입력에 **명령형 동사나 구체적 요구사항이 하나라도 있으면 스펙/조사 모드**로 기울여요. 사용자가 "구체적 계획을 달라" "조사해 달라"고 했는데 빈 PR 만 만드는 게 이 skill 의 가장 흔한 실패라서, 의심스러우면 스펙/조사 모드를 택해요.

### 1. 브랜치명 결정

`agent.md` 의 브랜치 명명 규칙(`<type>/<short-description>`) 을 그대로 따라요. (0단계에서 이미 결정됐다면 이 단계는 건너뜀.)

**type 추론** — 한국어 동사/명사에서 의도 파악:

| 입력 단서 | type |
|---|---|
| 추가 / 구현 / 만들어 / 신규 / 새 기능 | `feat` |
| 버그 / 수정 / 오류 / 고침 / 안 됨 | `fix` |
| 문서 / README / 주석 / 가이드 | `docs` |
| 리팩토링 / 정리 / 구조 변경 | `refactor` |
| 테스트 추가 / 커버리지 | `test` |
| 성능 / 최적화 | `perf` |
| 빌드 / 의존성 | `build` |
| CI / 워크플로우 / 액션 | `ci` |
| 그 외 잡일 / 설정 / 환경 | `chore` |

명확하지 않으면 `feat` 로 시작해요.

**description 생성** — kebab-case 영문 50자 이내:

- 한글이면 영어로 번역. 핵심 명사구만 남기고 관용어·조사·동사 제거.
- `포인트 결제 환불 기능 추가` → `point-payment-refund`
- `fcm ios 알림 사운드 버그 수정` → `fcm-ios-sound`
- `supplier query handler 리팩토링` → `supplier-query-handler`
- 입력에 GitHub 이슈 번호(`#123`, `이슈 123`, `344-...`)가 있으면 description 앞에 붙여요: `feat/344-kiosk-cafe-order-api`
- 패키지명을 굳이 description 에 넣지 않아요. scope 는 커밋 메시지가 표현해요.

**최종 브랜치명**: `{type}/{description}` (또는 `{type}/{issue-number}-{description}`)

### 2. 베이스 브랜치 동기화 확인

`origin/main` 의 최신을 받지 않은 상태에서 워크트리를 만들면 PR 이 부실하니, 먼저 한 번 fetch:

```bash
git fetch origin main
```

### 3. 워크트리 + 브랜치 생성

`EnterWorktree` 도구를 사용해 새 워크트리·브랜치를 한 번에 생성. 스키마는 `ToolSearch query="select:EnterWorktree"` 로 로드.

`EnterWorktree` 는 `.claude/worktrees/{name}/` 에 워크트리를 만들고 `worktree-{name}` 브랜치를 생성해요. 그래서 두 단계로 나눠 처리해요:

1. `EnterWorktree(name="{description}")` 로 격리된 작업 공간 진입
2. 워크트리 내부에서 1단계의 최종 브랜치명으로 rename:

```bash
git branch -m {type}/{description}
```

`EnterWorktree` 가 실패하면 fallback:

```bash
git worktree add -b {type}/{description} .claude/worktrees/{description} origin/main
cd .claude/worktrees/{description}
```

### 4. 빈 커밋

두 모드 모두 코드는 손대지 않으므로 빈 커밋 1개를 둬요. 조사·설계 결과는 파일이 아니라 PR body 로 들어가니까 여기서도 빈 커밋이면 충분해요.

```bash
git commit --allow-empty -m "$(cat <<'EOF'
{type}: {한국어 이유 요약}

작업 시작용 빈 커밋이에요. 실제 변경은 후속 커밋에서 들어가요.
EOF
)"
```

- `{type}` 은 1단계에서 결정한 값.
- 요약은 입력 `{이유}` 를 자연스럽게 정리한 한국어 한 문장 (50자 이내 권장).
- AI 공동저자 트레일러는 빈 커밋에는 붙이지 않아요 — 실제 변경이 없으니 공동저자 의미가 없어요.
- 스펙/조사 모드여도 커밋 메시지는 동일해요. 산출물은 6단계 PR body 에서 채워요.

### 5. 푸시

```bash
git push -u origin {type}/{description}
```

### 6. Draft PR 생성

0.5단계에서 가른 모드에 따라 PR body 가 달라져요. 두 경우 모두 `gh pr create --draft`, PR 제목은 conventional commit 헤더와 동일(`{type}: ...`). scope 는 부트스트랩 시점에 모르면 생략하고 후속 커밋에서 채워요.

#### 6-A. 부트스트랩 모드 — 골격 PR

빈 부트스트랩이라는 점을 반영해 짧게:

```bash
gh pr create --draft --base main --title "{type}: {한국어 이유 요약}" --body "$(cat <<'EOF'
## 개요

{한국어 이유 한 문단}.

> 작업 시작용 부트스트랩 PR 이에요. 실제 변경은 후속 커밋에서 들어가요.

## 작업사항

- [ ] (TBD)

## 관련 이슈

{이슈 번호가 있으면 `- 해결할 이슈: #{번호}`, 없으면 줄 자체를 생략}

## 테스트 방법

(작업 진행 후 채울 예정)

## 체크리스트

- [ ] 내 코드가 로컬 환경에서 정상적으로 동작하는지 확인했습니다.
- [ ] 기존 테스트가 성공적으로 완료되었습니다.
- [ ] 새로운 테스트를 추가하여 기능을 검증했습니다.
EOF
)"
```

#### 6-B. 스펙/조사 모드 — 산출물을 담은 PR

PR body 를 채우기 **전에** 실제 작업을 먼저 해요. 빈 골격만 올리면 안 돼요 — 이게 이 skill 의 핵심 동작이에요.

**B-1. 조사·설계 수행**

워크트리 안에서 관련 코드베이스를 실제로 읽고 조사해요. 규모가 있으면 `Explore` / `Plan` 에이전트에 위임해서 병렬로 훑어요:

- 구현 spec 요청이면 → 영향받는 파일·모듈·기존 패턴을 찾고, 변경 지점·인터페이스·단계별 작업 순서·엣지 케이스·테스트 전략을 정리해요.
- 조사 요청이면 → 질문에 답하기 위해 코드/문서/이슈를 읽고, 근거(파일 경로·라인, 외부 출처 URL)와 함께 결론을 정리해요. 필요하면 `WebSearch`/`WebFetch` 도 써요.
- 비교·의사결정 요청이면 → 후보안을 나열하고 장단점·트레이드오프·추천안을 정리해요.

추측으로 채우지 말고, 실제로 읽고 확인한 내용만 적어요. 확인 못 한 건 "확인 필요"로 명시.

**B-2. 결과를 PR body 로 작성**

요청 성격에 맞는 구조로 PR body 를 채워요. 아래는 구현 spec 예시 — 조사 요청이면 `## 구현 계획` 대신 `## 조사 결과`/`## 결론`/`## 근거` 같은 섹션으로 바꿔요:

```bash
gh pr create --draft --base main --title "{type}: {한국어 이유 요약}" --body "$(cat <<'EOF'
## 개요

{무엇을 / 왜 하는지 한 문단}.

## 배경 · 현재 동작

{조사로 파악한 현재 코드/동작. 근거가 되는 파일은 `path/to/file.ts:42` 형태로 명시}

## 구현 계획

1. {단계 1 — 건드릴 파일·함수, 어떻게 바꿀지}
2. {단계 2}
3. ...

## 고려사항 · 엣지 케이스

- {주의할 점, 의존성, 마이그레이션, 호환성}

## 작업사항

- [ ] {단계 1}
- [ ] {단계 2}

## 관련 이슈

{이슈 번호가 있으면 `- 해결할 이슈: #{번호}`, 없으면 줄 자체를 생략}

## 테스트 방법

{검증 시나리오 / 추가할 테스트}

## 체크리스트

- [ ] 내 코드가 로컬 환경에서 정상적으로 동작하는지 확인했습니다.
- [ ] 기존 테스트가 성공적으로 완료되었습니다.
- [ ] 새로운 테스트를 추가하여 기능을 검증했습니다.
EOF
)"
```

body 가 길어 셸 here-doc 이 부담되면, 임시 파일에 작성한 뒤 `gh pr create --body-file {파일}` 로 올리고 임시 파일은 지워요.

## 출력

작업 완료 후 사용자에게 다음 4가지를 보고:

1. 생성된 브랜치명
2. 워크트리 경로 (cd 해서 작업할 수 있도록)
3. 빈 커밋 SHA
4. Draft PR URL
5. 스펙/조사 모드였다면, PR body 에 담은 산출물 한 줄 요약 (예: "구현 계획 4단계 + 엣지 케이스 정리", "원인 조사 결과 PR body 에 정리")

예시:

```
브랜치  : feat/point-payment-refund
워크트리: /Users/.../monorepo/.claude/worktrees/point-payment-refund
커밋    : a1b2c3d4 (empty)
PR      : https://github.com/.../pull/123 (draft)
산출물  : 구현 spec 4단계 + 엣지 케이스를 PR body 에 작성
```

## 주의

- 이미 같은 이름의 브랜치/워크트리가 있으면 멈추고 사용자에게 알려요 — 덮어쓰지 않아요.
- `main` / `master` 브랜치 위에서 직접 실행돼도 EnterWorktree 가 격리해 주므로 사용자의 현재 작업 트리를 건드리지 않아요.
- **실제 코드 변경(파일 수정·구현)은 두 모드 모두 이 skill 의 책임이 아니에요.** 빈 커밋 + PR 까지만. 스펙/조사 모드에서도 산출물은 코드가 아니라 PR body 에 들어가요.
